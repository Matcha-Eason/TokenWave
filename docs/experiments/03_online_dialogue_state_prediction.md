# 桌宠状态预测实验

本实验是对前一轮“多通道波形基实验”的目标重做。目标不再是预测完整未来波形，而是改成桌宠插件更直接可用的两个问题：

1. 当前任务还会持续多久。
2. 当前或近期是否会进入长输出、长思考、反复测试和错误反馈组成的高负载状态。

## 输入

输入文件：

```text
outputs/coderforge_full/turn_timeseries.parquet
outputs/waveform_basis_experiment/nmf_basis.npy
```

逐轮时间序列表继续使用以下观测通道：

| 中文名 | 字段 | 用途 |
|---|---|---|
| 模型输出长度 | `assistant_tokens` | 判断长输出、输出压力和输出累积量 |
| 思考长度估计 | `thinking_tokens_est` | 判断规划/推理压力 |
| 环境反馈长度估计 | `user_tokens_est` | 判断工具返回、报错上下文和外部反馈压力 |
| 编辑次数 | `edit_count` | 判断任务推进和代码修改密度 |
| 测试次数 | `test_count` | 判断验证/调试强度 |
| 错误反馈次数 | `error_feedback_count` | 判断红温调试压力 |

工具调用次数没有作为主信号使用，因为前面的数据检查显示它几乎每轮固定为 1，信息量较低。

## 方法

入口脚本：

```text
scripts/pet_state_experiment.py
```

正式实验命令：

```bash
MPLCONFIGDIR=.matplotlib \
PYTHONPATH=scripts \
LOKY_MAX_CPU_COUNT=8 \
/Users/gyc/.cache/codex-runtimes/codex-primary-runtime/dependencies/python/bin/python3 \
scripts/pet_state_experiment.py \
  --input outputs/coderforge_full/turn_timeseries.parquet \
  --basis outputs/waveform_basis_experiment/nmf_basis.npy \
  --out-dir outputs/pet_state_experiment \
  --max-trajectories 30000 \
  --max-examples 120000
```

每条轨迹抽取若干个在线前缀样本。模型只能看到当前轮及之前的信息，包括：

- 当前轮序号。
- 最近 5 轮的输出、思考、环境反馈、错误、测试、编辑统计。
- 当前为止的累计输出、思考、环境反馈、错误、测试、编辑统计。
- 用已有 NMF 波形基对当前前缀拟合得到的形状权重。

刻意排除以下泄漏字段：

- 总轮数。
- 真实归一化进度。
- 未来标签。
- reward。
- 当前样本真实剩余轮数。

因此这次结果更接近桌宠在线运行时能拿到的信息。

## 预测目标

### 剩余交互长度

本实验同时做两个判断：

| 中文目标 | 机器学习形式 | 说明 |
|---|---|---|
| 剩余轮数估计 | 回归 | 估计当前之后还会有多少轮 |
| 5 轮内结束概率 | 二分类 | 判断是否已经进入明显收束阶段 |

对于桌宠来说，5 轮内结束概率比精确剩余轮数更实用。精确轮数受任务长尾影响明显，而“快结束/还要忙一阵”的状态更稳定。

### 高负载工作预警

本实验做两个短期预警：

| 中文目标 | 机器学习形式 | 标签定义 |
|---|---|---|
| 未来 3 轮长输出 | 二分类 | 未来 3 轮内是否出现 `assistant_tokens >= 250` |
| 未来 5 轮红温调试 | 二分类 | 未来 5 轮错误反馈和测试压力是否明显升高 |

这两个目标分别对应桌宠的“认真长时间输出”和“调试压力升高”。

### 桌宠状态

第一版状态机使用 5 个状态：

| 状态 | 英文字段 | 定义 |
|---|---|---|
| 读题理解 | `reading_understanding` | 早期阶段，尚未进入强输出、编辑、测试或错误反馈 |
| 稳定工作 | `steady_work` | 正常推进，负载中等 |
| 深度输出 | `deep_output` | 当前或最近 5 轮存在长输出压力 |
| 红温调试 | `overheat_debugging` | 最近 5 轮错误反馈和测试压力较高 |
| 收束 | `closing` | 真实标签定义为预计 3 轮内结束，模型预测时不能看到总轮数 |

状态标签是启发式标签，不是人工标注。它的作用是先验证这些桌宠状态是否能被当前数据中的在线特征稳定识别。

## 可解释性

特征重要性和目标基本一致。

剩余轮数主要依赖：

- 累计编辑次数。
- 累计测试次数。
- 当前轮序号。
- 累计思考长度。
- 最近编辑/测试密度。

这说明模型在学习“任务已经推进了多少实际工作”，而不是只看 token 长度。

未来长输出主要依赖：

- 累计模型输出长度。
- 当前轮序号。
- 累计环境反馈长度。
- 最近输出波动。
- 最近思考长度变化。

这说明长输出具有一定惯性：已经进入大输出/大上下文的轨迹，短期继续长输出的概率更高。

红温调试主要依赖：

- 累计错误反馈次数。
- 累计环境反馈长度。
- 最近错误反馈密度。
- 累计测试次数。
- 当前轮序号。

这符合“错误反馈 + 测试验证”组成调试压力的设定。

5 状态桌宠分类主要依赖：

- 最近错误反馈。
- 最近模型输出峰值。
- 当前轮序号。
- 累计测试和累计错误反馈。
- 累计环境反馈和累计模型输出。
- NMF 前缀波形基权重。

这说明 NMF 波形形状特征有辅助价值，但第一版桌宠状态更主要依赖可直接观测的近期事件和长度信号。

## 当前结论

| 问题 | 结论 |
|---|---|
| 是否应该继续预测完整未来波形 | 不应作为第一目标。完整波形可解释，但在线预测弱。 |
| 是否能预测剩余长度 | 可以，但精确轮数误差较大；更适合预测“是否快结束”和“短/中/长”区间。 |
| 是否能预测长输出 | 可以。未来 3 轮长输出 AUC 约 0.86，适合做桌宠“认真工作”预警。 |
| 是否能预测红温调试 | 有中等信号。AUC 约 0.72，适合做保守提示，不适合强断言。 |
| 5 状态桌宠分类是否可行 | 可行。启发式标签下宏平均 F1 约 0.92，但仍需要在真实 Codex 日志上校准。 |

## 对桌宠插件的建议

第一版桌宠插件不要输出一个单一的“对话进度百分比”。更合适的输出是：

```text
当前状态：读题理解 / 稳定工作 / 深度输出 / 红温调试 / 收束
5 轮内结束概率：0.0-1.0
未来 3 轮长输出概率：0.0-1.0
未来 5 轮红温概率：0.0-1.0
剩余轮数估计：只做内部参考，前端显示为短 / 中 / 长
```

桌宠表现可以这样映射：

| 模型输出 | 桌宠表现 |
|---|---|
| 读题理解 | 安静阅读、观察、轻微思考 |
| 稳定工作 | 正常敲代码、平稳工作 |
| 深度输出 | 专注输出、持续打字、认真工作 |
| 红温调试 | 高压调试、反复尝试、温度升高 |
| 收束 | 放松、整理、等待用户确认 |

## 重要限制

1. 当前状态标签是启发式标签，不是人工标注。
2. 数据来自 CoderForge 轨迹，不等价于真实 Codex 桌面插件日志。
3. `assistant_tokens >= 250` 是第一版长输出阈值，后续应按真实 Codex 日志重新校准。
4. 红温标签依赖错误反馈和测试事件；如果真实插件拿不到这些事件，需要降级为“环境反馈长度 + 失败关键词 + 重试次数”。
5. NMF 权重来自已有离线波形基，部署时需要把这个基作为预训练参数带入，而不是在线重新训练。

## 下一步

1. 增加“剩余长度区间分类”：短、中、长，比精确剩余轮数更贴近桌宠。
2. 在真实 Codex 日志上重建同样的在线前缀样本，检查阈值是否漂移。
3. 把模型导出成轻量推理模块，输入逐轮统计，输出状态和三个概率。
4. 为桌宠增加状态平滑，避免状态每轮来回跳变。
5. 后续再考虑把 NMF/谐波特征作为辅助解释层，而不是第一版核心判断。
